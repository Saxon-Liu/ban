## 三、安全分析（含务实评估）

> **重要前提**：本项目是 Electron 桌面应用，完全离线运行，无后端服务器。以下分析区分"理论风险"和"实际影响"，避免过度设计。

### 3.1 Electron 安全配置（已到位）

| 配置项 | 状态 | 说明 |
|--------|------|------|
| `nodeIntegration` | ✅ `false` | 渲染进程无法访问 Node.js API |
| `contextIsolation` | ✅ `true` | 渲染进程和 preload 脚本隔离 |
| IPC 通道白名单 | ✅ 已实现 | 仅暴露 3 个安全通道 |
| 外部请求拦截 | ✅ 全部阻止 | 非 `file://` 协议全部 `cancel: true` |
| CSP 策略 | ⚠️ 有 `unsafe-inline/eval` | 离线场景下风险可控 |
| 单实例锁 | ✅ 已启用 | 防止多开 |

### 3.2 前端认证机制

**密码存储和验证流程**：

```
constants.ts  →  默认密码 '0519'（硬编码）
Login.vue     →  localStorage 自定义密码（明文）
Login.vue     →  用户输入 === getCorrectKey()（明文比对）
main.ts       →  localStorage 'auth-token' === 'authenticated'（路由守卫）
```

**理论风险**：
- 密码硬编码在源码中，打包后 JS 文件可直接查看
- 认证 token 是固定字符串 `'authenticated'`
- 无暴力破解防护
- 恢复码 `519519` 硬编码在前端

**务实评估**：

| 风险 | 理论风险 | 实际影响 | 是否需要修复 |
|------|---------|---------|-------------|
| 硬编码密码 | 高 | **低** — 纯前端无法真正隐藏任何字符串，移除硬编码只是换位置，打包后仍在 JS bundle 中 | ❌ 不必要 |
| 密码哈希（bcrypt 等） | 高 | **无** — 哈希算法和盐值暴露在前端，攻击者可逆向整个流程。密码哈希的核心价值是"防止数据库泄露后的密码还原"，但此场景无数据库 | ❌ 不必要 |
| JWT Token | 高 | **无** — JWT 需要服务端签名验证，前端自签等于自己给自己开证明，任何人都可以伪造 | ❌ 不必要 |
| auth-token 固定值 | 中 | **低** — 可被 DevTools 绕过，但桌面应用场景下攻击面小 | ⚠️ 可选改进 |
| 修改密码不验证当前密码 | 中 | **中** — 如果他人趁你已登录时操作，可直接改掉密码（详见下方分析） | ✅ 建议做 |
| 恢复码硬编码 | 中 | **低** — 同"硬编码密码"，前端无法隐藏 | ❌ 不必要 |

### 3.3 实际建议做的安全改进

| 改进项 | 成本 | 价值 | 说明 |
|--------|------|------|------|
| **修改密码时验证当前密码** | 低 | 中 | 防止他人趁已登录时篡改密码（详见下方） |
| **auth-token 改为随机值** | 低 | 低 | 防止最简单的 `setItem` 绕过 |
| **前端输入验证加强** | 低 | 低 | 文件导入增加数据类型和范围校验 |

#### 修改密码不验证当前密码 — 详细分析

**位置**: `src/pages/ConfigManagement.vue` L137-L200

**当前流程**：

```
用户点击"修改密码" → 弹窗仅要求输入新密码 + 确认新密码 → 直接 localStorage.setItem(CUSTOM_KEY_STORAGE, newPassword)
```

**问题**：`handleChangePassword` 函数（L179-L200）直接写入新密码，没有要求输入当前密码：

```typescript
const handleChangePassword = async () => {
  // ... 表单验证（只验证新密码和确认密码）
  changingPassword.value = true
  try {
    localStorage.setItem(CUSTOM_KEY_STORAGE, passwordForm.newPassword)  // 直接写入，无当前密码验证
    ElMessage.success('密码修改成功，下次登录请使用新密码')
  }
}
```

**当前表单验证**（L149-L174）：
- `validateNewPassword`：校验非空、长度≥4、不能与当前密码相同
- `validateConfirmPassword`：校验两次输入一致

**缺少的验证**：当前密码输入框 + 与 `getCorrectKey()` 比对

**实施方案**：在表单中增加"当前密码"字段，验证通过后才允许修改

```typescript
// 表单增加当前密码字段
const passwordForm = reactive({
  currentPassword: '',  // ← 新增
  newPassword: '',
  confirmPassword: '',
})

// 增加当前密码验证
const validateCurrentPassword = (_rule: any, value: string, callback: any) => {
  if (!value) {
    callback(new Error('请输入当前密码'))
  } else if (value !== getCorrectKey()) {
    callback(new Error('当前密码不正确'))
  } else {
    callback()
  }
}

// handleChangePassword 中无需额外逻辑，表单验证已确保当前密码正确
```

### 3.4 不建议做的安全改造（过度设计）

| 方案 | 不建议原因 |
|------|-----------|
| 移除硬编码密码 | 打包后密码仍在 JS bundle 中，只是换了位置 |
| 密码哈希存储 | 哈希算法暴露在前端，任何人可逆向 |
| 引入 JWT | 没有服务端签名验证，前端自签 JWT 毫无意义 |
| 添加暴力破解防护 | 桌面应用、团队内部使用、用户量小，投入产出比低 |
| Content Security Policy | Electron 主进程已配置 CSP 且拦截所有外部请求 |

### 3.5 安全总结

> 对于一个**内部使用的 Electron 桌面应用**，当前安全配置已经足够。密码保护的本质是"防君子不防小人"，目的是防止同事误操作或随意进入，而非抵御恶意攻击。如果未来需要暴露到公网或有多租户需求，才需要引入后端 + 真正的认证体系。

## 四、性能分析（含务实评估）

### 4.1 操作后全量数据刷新 — ✅ 建议优化

**位置**: `src/pages/views/ScheduleTableView.vue`

**问题**：每次排班操作后都要等待 IndexedDB 查询 + 响应式更新 + 重新渲染整个视图。

**`loadData()` 全量刷新调用点**（共 4 处，其中 3 处可优化）：

| # | 函数 | 行号 | 触发场景 | 是否可优化 |
|---|------|------|---------|-----------|
| 1 | `handleCellDrop` | L748 | 整格拖拽复制/移动成功后 | ✅ 改为乐观更新 |
| 2 | `handleCellDrop` | L812 | 人员拖入格子排班成功后 | ✅ 改为乐观更新 |
| 3 | `handleTagReorder` | L841 | 同单元格内拖拽排序后 | ✅ 改为乐观更新 |
| 4 | `removeSchedule` | L964 | 删除排班记录后 | ✅ 改为乐观更新 |
| — | `onMounted` | L974 | 页面初始化 | ❌ 保留（首次必须全量加载） |
| — | `watch(currentMonth)` | L981 | 切换月份 | ❌ 保留（月份切换必须全量加载） |

**`loadData()` 全量加载内容**（L422-L458）：

```typescript
const loadData = async () => {
  // 1. 人员全量查询 — 大部分操作不需要重新查
  const peopleData = await repositories.people.getAllIncludingArchived()
  // 2. 班次全量查询 — 几乎不会变
  const shiftsData = await repositories.shifts.getAllIncludingArchived()
  // 3. 当月排班查询 — 只有这条在排班操作后可能需要
  const schedulesData = await repositories.schedules.getByMonth(currentMonth.value)
  // 4. 额外休息配置 — 几乎不会变
  const config = await repositories.extraRestConfigs.getByYearAndMonth(year, monthNum)
}
```

**务实评估**：这是**用户体验提升最大的优化点**。操作后全量刷新会让用户感受到明显的卡顿感，改用乐观更新（先更新内存数据、立即渲染、异步持久化）可以将操作响应时间从"可感知的等待"变为"即时反馈"。

**实施方案**：以 `removeSchedule` 为例，当前是：

```typescript
// 当前（全量刷新）
const removed = await scheduleService.removeScheduleByIdentity(personId, date, shiftId)
if (removed) {
  ElMessage.success("删除排班成功")
  await loadData()  // 全量刷新 4 个表
}
```

优化后：

```typescript
// 优化（乐观更新）
const removed = await scheduleService.removeScheduleByIdentity(personId, date, shiftId)
if (removed) {
  schedules.value = schedules.value.filter(
    s => !(s.personId === personId && s.date === date && s.shiftId === shiftId)
  )  // 立即更新内存，Vue 响应式触发重渲染
  ElMessage.success("删除排班成功")
}
```

**注意**：`PersonCalendarView.vue` 的排班操作较少（日历视图主要用于查看，排班在表格视图完成），`loadData` 仅在 `onMounted` 调用一次，暂不需要优化。

### 4.2 IndexedDB 查询效率 — ✅ 建议优化

**位置**: `src/repositories/IndexedDBScheduleRepository.ts` L97-L101

```typescript
async getByPersonAndMonth(personId: string, month: string): Promise<Schedule[]> {
  const db = await dbManager.getDB()
  const personSchedules = await db.getAllFromIndex('schedules', 'by-personId', personId)
  return personSchedules.filter(schedule => schedule.month === month)  // 先取全部月份，再内存过滤
}
```

**问题**：`getByPersonAndMonth` 先通过 `by-personId` 索引取该人员的**全部历史排班**（所有月份），再在内存中过滤出目标月份。随着使用时间增长，历史数据越多，查询越慢。

**当前 schedules 表索引**（`IndexedDBManager.ts` L77-L88）：

| 索引名 | 字段 | 类型 |
|--------|------|------|
| `by-personId` | `personId` | 单字段 |
| `by-shiftId` | `shiftId` | 单字段 |
| `by-date` | `date` | 单字段 |
| `by-month` | `month` | 单字段 |
| `by-personId-date` | `[personId, date]` | 复合（unique） |

**实施方案**：添加 `by-personId-month` 复合索引 + 升级数据库版本

```typescript
// IndexedDBManager.ts — 需要 3 处改动：

// 1. 版本号从 2 改为 3
const DB_CONFIG = {
  NAME: "ScheduleManagementDB",
  VERSION: 3,  // ← 原为 2
} as const;

// 2. upgrade 回调中添加兼容旧版本的升级逻辑
if (oldVersion < 3) {
  // 旧版本可能没有复合索引，需要重建 schedules 表
  const schedulesStore = db.transaction('schedules').objectStore('schedules');
  if (!schedulesStore.indexNames.contains('by-personId-month')) {
    schedulesStore.createIndex('by-personId-month', ['personId', 'month']);
  }
}
// 注意：idb 库的 upgrade 回调中无法直接获取已存在的 store 进行 addIndex，
// 需要处理版本兼容。最简单的方案是：

// 3. getByPersonAndMonth 改用复合索引查询
async getByPersonAndMonth(personId: string, month: string): Promise<Schedule[]> {
  const db = await dbManager.getDB()
  return db.getAllFromIndex('schedules', 'by-personId-month', [personId, month])
}
```

**务实评估**：改动极小（几行代码），但查询性能提升显著，特别是当系统使用时间越长、历史数据积累越多时。数据库版本升级需要处理旧数据兼容，建议参考 `oldVersion < 2` 的处理模式。

### 4.3 batchUpdate N+1 查询 — ✅ 建议优化

**位置**: `src/repositories/IndexedDBScheduleRepository.ts` L185-L206

```typescript
async batchUpdate(updates: { id: string; data: UpdateData<Schedule> }[]): Promise<Schedule[]> {
  const db = await dbManager.getDB()
  const updatedSchedules: Schedule[] = []
  
  for (const { id, data } of updates) {
    const schedule = await this.getById(id)  // 每条都单独查询一次
    if (!schedule) {
      throw new Error(`排班记录不存在: ${id}`)
    }
    const updatedSchedule: Schedule = {
      ...schedule,
      ...data,
      updatedAt: getCurrentDateTime(),
    }
    await db.put('schedules', updatedSchedule)  // 每条都单独写入一次
    updatedSchedules.push(updatedSchedule)
  }
  return updatedSchedules
}
```

**问题**：每条更新都先 `getById` 再 `put`，每次操作都是独立的 IndexedDB 事务。拖拽排序 10 条记录产生 20 次事务，30 条产生 60 次事务。

**调用场景**：`batchUpdate` 被 `scheduleService.ts` 中的两个函数调用：
- `insertScheduleIntoCell`（L178）— 插入排班时重排序号
- `reorderSchedulesInCell`（L205）— 拖拽排序时更新序号

**实施方案**：改为单事务批量读写

```typescript
async batchUpdate(updates: { id: string; data: UpdateData<Schedule> }[]): Promise<Schedule[]> {
  const db = await dbManager.getDB()
  const tx = db.transaction('schedules', 'readwrite')
  const store = tx.store
  const updatedSchedules: Schedule[] = []
  
  // 批量读取
  const existingList = await Promise.all(
    updates.map(({ id }) => store.get(id))
  )
  
  // 批量写入
  const now = getCurrentDateTime()
  for (let i = 0; i < updates.length; i++) {
    const existing = existingList[i]
    if (!existing) {
      throw new Error(`排班记录不存在: ${updates[i].id}`)
    }
    const updated = { ...existing, ...updates[i].data, updatedAt: now }
    store.put(updated)
    updatedSchedules.push(updated)
  }
  
  await tx.done
  return updatedSchedules
}
```

**务实评估**：改为单事务批量读写的改动成本低，排序操作（特别是拖拽排序多条记录时）体感提升明显。所有读写在同一个事务中完成，IndexedDB 只做一次磁盘提交。

#### 排序逻辑重复详情（6 处，分布在 5 个文件中）

所有排序逻辑都是"先按 `order` 排序，再按 `createdAt` 排序"，但实现细节略有差异：

| # | 文件 | 行号 | 函数/位置 | 默认值 | 日期处理 | 差异 |
|---|------|------|----------|--------|---------|------|
| 1 | `scheduleService.ts` | L42-L51 | `sortSchedulesByOrder` | `?? 0` | `Date.getTime()` | 标准实现 |
| 2 | `ScheduleTableView.vue` | L404-L413 | `schedulesByCell` 计算属性 | `?? 0` | `getTime?.()` 防御 | 有可选链防御 |
| 3 | `PersonCalendarView.vue` | L434-L438 | `loadBaseData` 内部 | `?? 999` | 无 createdAt 排序 | **默认值不同（999 vs 0）** |
| 4 | `IndexedDBPeopleRepository.ts` | L17-L23 | `sortPeople` 私有方法 | `typeof` 判断 | `Number(new Date())` | **回退逻辑不同** |
| 5 | `IndexedDBShiftRepository.ts` | L16-L22 | `sortShifts` 私有方法 | `typeof` 判断 | `Number(new Date())` | **回退逻辑不同** |
| 6 | `excelExport.ts` | L80-L94 | Excel 导出分组排序 | `?? 0` | `any` 类型防御 | 多层类型防御 |

**实施方案**：在 `src/utils/common.ts` 中抽取统一的排序工具函数

```typescript
/**
 * 按 order 升序排序，order 相同时按 createdAt 升序
 */
export function sortByOrder<T extends { order?: number; createdAt?: Date | string }>(
  items: T[]
): T[] {
  return [...items].sort((a, b) => {
    const ao = a.order ?? 0
    const bo = b.order ?? 0
    if (ao !== bo) return ao - bo
    const at = a.createdAt instanceof Date ? a.createdAt.getTime() : new Date(a.createdAt).getTime()
    const bt = b.createdAt instanceof Date ? b.createdAt.getTime() : new Date(b.createdAt).getTime()
    return at - bt
  })
}
```

然后将 6 处排序代码统一替换为 `sortByOrder(xxx)` 调用。

#### #7 无操作自动退出登录 — 详细需求分析

**需求描述**：用户在登录状态下，若连续 5 分钟无任何操作，自动退出登录并跳转到登录页。其中前 2 分钟静默计时，2 分钟时弹窗提醒，用户有 3 分钟时间决定是否继续使用。

**交互流程**：

```
用户登录成功
  │
  ├─ 静默计时 2 分钟（监听用户操作事件）
  │    │
  │    ├─ 用户有操作 → 重置计时器，重新开始 2 分钟静默计时
  │    │
  │    └─ 2 分钟无操作 → 弹出提醒弹窗
  │         │
  │         ├─ 用户点击"继续使用" → 关闭弹窗，重置计时器，重新开始 2 分钟静默计时
  │         ├─ 用户点击"退出登录" → 立即退出登录
  │         ├─ 用户有操作（鼠标/键盘/滚动） → 自动关闭弹窗，重置计时器
  │         │
  │         └─ 3 分钟无响应（总计 5 分钟） → 自动退出登录，跳转登录页
  │
  └─ 切换月份/页面等路由操作 → 视为有操作，重置计时器
```

**"无操作"的事件定义**：

| 事件 | 是否监听 | 说明 |
|------|---------|------|
| `mousedown` | ✅ | 鼠标点击，排除纯移动 |
| `keydown` | ✅ | 键盘输入 |
| `scroll` | ✅ | 页面滚动 |
| `touchstart` | ✅ | 触摸操作（触屏设备兼容） |
| `mousemove` | ❌ | 不监听，鼠标微小移动容易误触导致计时器频繁重置 |

**退出登录的处理**：
1. 清除 `localStorage` 中的 `AUTH_STORAGE_KEY` 认证状态
2. 清除 `localStorage` 中的 `AUTH_EXPIRY_KEY` 过期时间
3. 关闭弹窗（如果正在显示）
4. 跳转到 `/login` 路由

**实施方案**：

```typescript
// src/composables/useAutoLogout.ts（新增文件）

import { ref, onMounted, onUnmounted } from 'vue'
import { useRouter } from 'vue-router'
import { ElMessageBox } from 'element-plus'
import { AUTH_STORAGE_KEY, AUTH_EXPIRY_KEY } from '@/utils/constants'

const SILENT_TIMEOUT = 2 * 60 * 1000    // 2 分钟静默期
const TOTAL_TIMEOUT = 5 * 60 * 1000     // 5 分钟总超时

export function useAutoLogout() {
  const router = useRouter()
  const warningShown = ref(false)
  let silentTimer: ReturnType<typeof setTimeout> | null = null
  let logoutTimer: ReturnType<typeof setTimeout> | null = null

  const clearAllTimers = () => { /* 清除两个定时器 */ }

  const doLogout = () => {
    clearAllTimers()
    ElMessageBox.close()  // 关闭可能存在的弹窗
    localStorage.removeItem(AUTH_STORAGE_KEY)
    localStorage.removeItem(AUTH_EXPIRY_KEY)
    router.push('/login')
  }

  const showWarning = async () => {
    warningShown.value = true
    // 设置 3 分钟后自动退出的定时器
    logoutTimer = setTimeout(doLogout, TOTAL_TIMEOUT - SILENT_TIMEOUT)
    try {
      await ElMessageBox.confirm(
        '您已长时间未操作，即将自动退出登录',
        '会话超时提醒',
        { confirmButtonText: '继续使用', cancelButtonText: '退出登录', type: 'warning' }
      )
      // 点击"继续使用"
      resetTimers()
    } catch {
      // 点击"退出登录"
      doLogout()
    }
  }

  const resetTimers = () => {
    clearAllTimers()
    warningShown.value = false
    silentTimer = setTimeout(showWarning, SILENT_TIMEOUT)
  }

  const onUserActivity = () => {
    if (warningShown.value) {
      // 弹窗期间有操作 → 自动关闭弹窗并重置
      ElMessageBox.close()
      resetTimers()
      return
    }
    resetTimers()
  }

  onMounted(() => {
    const events = ['mousedown', 'keydown', 'scroll', 'touchstart']
    events.forEach(e => document.addEventListener(e, onUserActivity, { passive: true }))
    resetTimers()
    // ...
  })

  onUnmounted(() => {
    clearAllTimers()
    // 移除事件监听
  })
}
```

**在 `App.vue` 中使用**：

```vue
<script setup lang="ts">
import { useAutoLogout } from '@/composables/useAutoLogout'
useAutoLogout()  // 全局挂载，自动监听
</script>
```

**注意事项**：
- 仅在用户已登录时启用（检查 `localStorage` 中的认证状态）
- 路由跳转到 `/login` 时应停止计时器，避免登录页也弹超时提醒
- 弹窗期间用户有任何操作应自动关闭弹窗并重置计时器，而非强制等用户点按钮
- 事件监听使用 `{ passive: true }` 选项，避免影响滚动性能

**涉及文件**：

| 文件 | 操作 | 说明 |
|------|------|------|
| `src/composables/useAutoLogout.ts` | 新增 | 自动退出核心逻辑 |
| `src/App.vue` | 修改 | 挂载 `useAutoLogout()` |
| `src/utils/constants.ts` | 可选修改 | 超时时间可提取为常量 |
