# TODO / 未来计划

本文档记录后续版本计划，并约束功能边界。

## 产品定位

这是一个单人、单电脑、离线优先的本地排班工具。

核心用途：

- 观察人员、班次和日期分布
- 快速试排、调整、比对
- 导出 Excel 作为参考
- 最终结果手动录入钉钉 OA 等正式系统

因此，本项目不追求替代 OA，不做多人协作，不做复杂审批，也不做强规则自动排班。后续功能应继续服务于“简单、快速、清楚、少出错”。

## 1. 首次使用创建密码 / 重置密码

### 背景

当前系统使用默认登录密码和默认恢复码，适合开发和初始测试，但正式使用时不够自然，也容易让用户忽略首次安全配置。

### 目标

- 首次打开系统时，引导用户主动创建登录密码。
- 不再要求用户先使用默认密码登录。
- 忘记密码时，可通过恢复码设置新密码。
- 修改密码或重置密码后，所有旧登录会话立即失效。
- 初始化系统时，密码策略要清晰可预期。

### 交互方案

- 登录页增加“首次设置”模式。
- 当本地未检测到自定义密码时，显示创建密码表单。
- 创建密码时要求二次确认。
- 创建成功后进入系统，或回到登录页重新登录。
- 忘记密码入口保留，文案从“恢复默认密码”逐步调整为“重置密码”。
- 重置密码优先支持“输入恢复码后设置新密码”。
- “恢复为默认密码”后续考虑隐藏或移除，减少默认密码依赖。

### 技术任务

- 增加本地密码初始化状态判断。
- 梳理这些存储键的职责边界：
  - `CUSTOM_KEY_STORAGE`
  - `AUTH_CREDENTIAL_VERSION_KEY`
  - `AUTH_SESSION_VERSION_KEY`
- 抽出统一密码写入逻辑，复用到首次设置、修改密码、忘记密码重置。
- 重置密码成功后调用会话失效逻辑。
- 更新 README 中默认密码和恢复码说明。

### 验收标准

- 新安装或清空数据后，首次进入会引导创建密码。
- 创建密码后可正常登录。
- 忘记密码后可通过恢复码设置新密码。
- 重置密码后，旧会话不能继续使用。
- 初始化系统时，如果选择重置密码，能回到首次设置或明确的默认密码策略。

## 2. 检查更新 / portable 热更新

### 背景

当前项目可以打包安装版、便携版和绿色版，但应用内没有更新检查能力。未来计划通过 GitHub Releases 发布更新包，尤其希望 portable 便携版也能在应用内完成更新，减少手动下载和替换 exe 的成本。

这里的“热更新”不是运行中替换代码，而是：

1. 应用内检查新版本。
2. 下载更新包。
3. 退出当前程序。
4. 替换 exe。
5. 自动重启新版。

### 分期计划

#### 阶段 1：检查更新

- 在“配置管理”或后续“关于”区域增加“检查更新”入口。
- 显示当前版本、最新版本、发布时间和更新说明。
- 有新版本时提供“打开下载页面”。
- 无网络或接口失败时，只提示失败，不影响使用。
- Web 模式下隐藏或禁用桌面更新能力。

#### 阶段 2：portable 下载更新包

- 识别当前运行形态是否为 portable。
- 从 GitHub Releases 下载新版 `ban-<version>-portable.exe`。
- 下载到 `userData/updates` 或系统临时目录。
- 下载过程显示进度、速度和剩余大小。
- 下载失败可重试，可取消。

#### 阶段 3：portable 退出替换重启

portable 单文件 exe 正在运行时，Windows 通常不允许直接覆盖当前 exe，因此需要独立 updater helper。

推荐流程：

1. 下载新版 portable exe。
2. 校验文件大小和 `sha256`。
3. 记录当前 exe 路径、下载文件路径、旧版备份路径。
4. 启动独立 updater helper。
5. 当前应用退出。
6. helper 等待当前进程完全退出。
7. helper 将旧 exe 改名为备份文件。
8. helper 将新版 exe 移动到原 exe 路径。
9. helper 启动新版 exe。
10. 新版启动成功后，延迟清理旧版备份。

### GitHub Releases 要求

- Release 中至少提供 portable 资源。
- 资源命名保持稳定：`ban-<version>-portable.exe`。
- 建议同时提供 `sha256` 校验信息。
- Release notes 写清楚变更内容、修复内容和兼容提醒。
- 后续如果同时支持 setup/green，需要能区分不同资源类型。

### 技术方案

- 主进程新增安全 IPC，例如：
  - `checkForUpdates`
  - `downloadUpdate`
  - `installPortableUpdate`
- 主进程请求 GitHub Releases API，或读取仓库中的版本 JSON。
- 返回结构化结果：
  - `currentVersion`
  - `latestVersion`
  - `hasUpdate`
  - `releaseDate`
  - `releaseNotes`
  - `downloadUrl`
  - `assetName`
  - `assetSize`
  - `sha256`
- 使用语义化版本比较，避免字符串比较误判。
- 网络请求必须有超时和错误兜底。
- 更新相关日志写入主进程日志，失败时只记录必要摘要。

### 风险与限制

- 不能在主进程里直接覆盖 `process.execPath`。
- helper 必须独立于当前应用进程。
- 如果程序位于 U 盘、只读目录或权限受限目录，自动替换可能失败。
- 未签名 exe 可能被 Windows Defender 或杀毒软件延迟扫描。
- 更新失败必须保留旧版 exe，不能让用户无法启动。
- 安装版自动更新涉及签名、安装器和回滚，放到更后续版本。

### 验收标准

- 桌面版能显示当前版本。
- 有新版本时能提示并打开 GitHub Release 下载页面。
- 无新版本时提示当前已是最新版。
- 无网络或接口异常时不会卡住页面。
- portable 版本能下载新版 exe。
- portable 更新失败时旧版仍可启动。
- 检查更新不会影响 Web 模式运行。

## 3. 撤销 / 操作历史

### 背景

当前排班以手动拖拽为主。真实场景里有临时请假、熟手搭配、临时调整等大量动态判断，自动排班难以形成稳定规则。相比自动生成方案，撤销和操作历史更贴近实际需求。

### 目标

- 支持撤销最近一次排班操作。
- 先覆盖排班页高频操作，不追求全系统完整时间旅行。
- 降低拖错、删错、移动错带来的返工成本。
- 撤销逻辑要可靠，不破坏 IndexedDB 中的真实数据。

### 优先覆盖的操作

- 新增排班
- 移动单个人员排班
- 同格排序
- 删除排班
- 整格移动
- 整格复制

### 交互方案

- 排班页增加撤销按钮，或在操作成功提示中提供“撤销”。
- 每次成功操作后记录一条操作快照。
- 有可撤销操作时按钮可用；没有操作时置灰。
- 撤销成功后提示“已撤销上一步操作”。
- 切换月份后清空撤销栈，避免跨月份状态复杂化。

### 技术方案

- 在排班页维护轻量 undo stack。
- 每条记录包含：
  - 操作类型
  - 操作前受影响的排班记录
  - 操作后新增、更新、删除的排班记录 ID
  - 当前月份
- 撤销时通过 `scheduleService` 或 repository 执行反向操作。
- 撤销失败时记录错误日志，并提示用户刷新后重试。

### 验收标准

- 新增排班后可以撤销，撤销后记录消失。
- 移动排班后可以撤销，撤销后回到原日期和原班次。
- 删除排班后可以撤销，撤销后恢复原记录。
- 整格复制后可以撤销，撤销后删除复制出来的记录。
- 整格移动后可以撤销，撤销后恢复来源格和目标格顺序。
- 切换月份后撤销按钮不会错误影响其他月份。

## 暂不做

以下功能暂不进入近期计划：

- 自动排班：当前排班依赖大量临时判断，例如请假、熟手搭配、人员状态，不适合抽象成稳定规则。
- 自动备份：当前定位为单人单机离线工具，已有手动导出配置；自动备份会增加恢复选择和文件管理成本。
- 多人协作 / 云同步：会明显改变本地优先架构。
- 复杂权限系统：当前不是多人在线系统。
- AI 排班：如果未来考虑，也只能作为辅助建议，不能替代人工判断。
- 钉钉 OA 自动录入：正式结果仍手动录入 OA，本工具不直接操作第三方系统。
- 复杂请假/技能规则引擎：短期只做观察和手动调整，不把临时经验规则化。

## 优先级建议

1. 首次使用创建密码：提升正式使用体验和安全感。
2. 忘记密码重置流程统一：和首次设置属于同一批认证体验优化。
3. 检查更新：先通过 GitHub Releases 获取版本信息，做“提醒 + 打开下载页”。
4. portable 热更新：下载新版 exe，退出后替换并重启，重点处理失败回滚。
5. 撤销 / 操作历史：贴合手动排班场景，减少误操作成本。
6. 安装版自动更新：涉及签名、安装器、失败回滚，放到更后续版本。
